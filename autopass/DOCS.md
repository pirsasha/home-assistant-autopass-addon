# Автопропуск

## Установка

1. Добавьте репозиторий add-on в Home Assistant.
2. Установите add-on **Автопропуск**.
3. Обязательно задайте уникальный `api_key` в конфигурации add-on.
4. Запустите add-on и включите отображение в боковой панели.

База SQLite хранится в `/data/autopass.db` внутри постоянного каталога add-on и включается в резервные копии Home Assistant.

Снимки событий Frigate автоматически копируются в `/data/snapshots`. Поэтому
они остаются доступны в журнале даже после очистки событий во Frigate.

## Локальный вход по IP-адресу

По умолчанию интерфейс открывается только через Home Assistant Ingress. Чтобы
разрешить отдельный вход для рабочего места охраны:

1. откройте страницу add-on **Конфигурация**;
2. включите `local_access_enabled`;
3. в разделе **Сеть** опубликуйте контейнерный порт `8099`, например как
   порт хоста `8099`;
4. перезапустите add-on;
5. откройте add-on через Home Assistant и в разделе
   **Настройки → Пользователи и права** создайте локального пользователя;
6. откройте `http://IP_HOME_ASSISTANT:8099` и войдите созданными данными.

Ingress и локальный вход работают одновременно. Пароли не сохраняются открытым
текстом: используется `scrypt` с индивидуальной солью. Локальная сессия действует
14 дней, хранится в `HttpOnly` cookie и завершается при смене пароля. После пяти
неверных попыток повторный вход блокируется на 15 минут.

Не публикуйте порт `8099` напрямую в интернет. Для внешнего доступа используйте
VPN или HTTPS reverse proxy. Обычный HTTP допустим только в доверенной локальной
сети.

## Индивидуальные права

Роли администратора, диспетчера, охранника и наблюдателя остаются готовыми
шаблонами. Администратор может отдельно включать и выключать каждому
пользователю:

- просмотр пропусков и журнала;
- создание любых либо только временных и одноразовых пропусков;
- удаление, импорт и экспорт;
- ручное открытие ворот;
- настройки, пользователей и аудит;
- парковочные сессии, оплату, тарифы и отчёты.

Изменения роли и индивидуальных разрешений записываются в аудит. Система не
позволяет удалить или лишить права управления пользователями последнего
администратора.

## Лицензия

После первой установки доступен демонстрационный период 14 дней и льготный
период 7 дней. ID установки отображается в разделе **О программе → Лицензия**.
Передайте этот ID поставщику и загрузите полученный JSON-файл в том же разделе.

Лицензия проверяется офлайн по цифровой подписи Ed25519. Закрытый ключ
поставщика не хранится в add-on. После завершения демо или лицензии нельзя
создавать, изменять и удалять пропуска, а автоматическое открытие отключается.
Журнал, экспорт, резервные копии и ручное аварийное открытие остаются доступны.

Команды выпуска лицензий описаны в `tools/README.md`.

Коммерческая парковка лицензируется отдельно. Для её включения файл лицензии
должен содержать оба модуля:

```json
"modules": ["autopass", "parking"]
```

В локальном веб-генераторе для этого включите флажок **Коммерческая парковка**.
Лицензия только с модулем `autopass` продолжает управлять пропусками и воротами,
но парковочный API возвращает `402`.

## О программе и обновления

В разделе **О программе** собраны:

- установленная и доступная версия add-on;
- состояние лицензии, ID установки, клиент, срок и подключённые модули;
- форма загрузки файла лицензии для администратора;
- контакты разработчика и поддержки.

Наличие обновления определяется по данным Supervisor Home Assistant. Само
обновление устанавливается на странице add-on в Home Assistant. Через локальный
вход по IP раздел также доступен, но загрузка лицензии показывается только
пользователю с разрешением управления настройками.

## Тарифы парковки

При активном модуле `parking` администратору доступна вкладка **Парковка**.
Тариф содержит:

- название и класс автомобиля;
- бесплатный период;
- цену каждого начатого часа после бесплатного периода;
- необязательный суточный максимум;
- время, за которое автомобиль должен выехать после оплаты;
- признак активности.

В интерфейсе цены вводятся в рублях, но в SQLite и API хранятся целым числом
копеек. Это исключает ошибки двоичного округления денежных значений.

Суточный максимум применяется к 24-часовым блокам от момента въезда, а не к
календарным дням. Например, если автомобиль въехал в 15:30, первый суточный
блок считается до 15:30 следующего дня; начатый следующий блок рассчитывается
отдельно. Бесплатный период применяется один раз в начале парковки. Управление
тарифами доступно только роли с разрешением `parking_tariffs`.

## Оплата и выезд

В активной парковочной сессии пользователь с правом кассира нажимает
**Оплатить**, выбирает способ оплаты и подтверждает сумму. Сумма рассчитывается
на момент оплаты и фиксируется в неизменяемом журнале вместе с пользователем
Home Assistant и временем операции.

После оплаты сессия получает срок разрешённого выезда из тарифа. Выездная камера
открывает настроенные выездные ворота только при совпадении номера и действующем
сроке. Сессия закрывается после успешной отправки команды ворот. Если срок
истёк, кассир принимает рассчитанную доплату и выдаёт новый срок выезда.

Если камера не прочитала грязный номер, оператор может закрыть разрешённую
сессию вручную. Администратор или кассир может выполнить принудительное
закрытие, но обязан указать причину; действие сохраняется в аудите.

## Кассовые смены и отчёты

Перед первым платежом кассир открывает смену и указывает начальный остаток
наличных. Все новые платежи получают номер этой смены. Текущая X-сводка
показывает количество платежей, общую выручку, наличные, карты и переводы.

При закрытии кассир вводит фактическую сумму наличных. Система фиксирует
ожидаемую сумму, фактическую сумму и расхождение. Закрытая Z-сводка больше не
пересчитывается при последующих операциях.

В разделе отчётности выбирается период. Доступны выручка по способам оплаты,
посещаемость, уникальные автомобили, средняя длительность, текущая загрузка и
журнал завершённых парковок. Реестр закрытых сессий выгружается в CSV или XLSX.
Доступ к отчётам требует разрешения `parking_reports`.

## Операционная устойчивость

В настройках коммерческой парковки задаются вместимость, порог предупреждения
и поведение при заполнении. Режим блокировки запрещает только новый платный
въезд. Автомобили с действующим пропуском продолжают въезжать. Оператор может
создать сессию сверх лимита, явно подтвердив исключение.

Сессии старше заданного количества часов помечаются как требующие проверки.
Оператор может найти такую сессию в активном списке и принудительно закрыть её
с обязательной причиной.

Настройка **Минимальное время между въездом и выездом** защищает от ситуации,
когда выездная камера видит задний номер автомобиля сразу после проезда через
въезд. По умолчанию автоматический выезд по номеру игнорируется первые
60 секунд. Событие сохраняется в журнале, парковочная сессия остаётся открытой,
а лимит использований пропуска не расходуется повторно. Значение `0` отключает
защиту. Настройка не ограничивает QR, ручное закрытие и аварийное открытие.

Дополнительно настройте во Frigate отдельные зоны въезда и выезда так, чтобы
выездная зона не захватывала участок сразу за въездным шлагбаумом. Программный
интервал является страховкой, а не заменой корректных зон камеры.

Кнопки аварийного открытия въезда и выезда выполняют реальную команду ворот и
всегда требуют причину. Пользователь, время, точка доступа и результат команды
сохраняются в аудите.

Физическое поведение ворот при полном отказе Home Assistant обеспечивает
контроллер ворот, реле и резервное питание. В add-on выбранная политика и
инструкция оператору хранятся как эксплуатационный регламент.

Для фотографий парковки настраивается отдельный срок хранения. Старые закрытые
сессии можно автоматически обезличивать: удаляются номер, талон и фотографии,
но сохраняются время, тариф, суммы оплаты и агрегированная статистика. Значение
`0` отключает автоматическое обезличивание.

## Каналы идентификации

Для въезда и выезда независимо выбирается режим:

- `Только номер`;
- `Номер + резервный QR`;
- `QR + проверка номера`;
- `Только QR`;
- `Номер + резервный звонок`;
- `Только оператор`.

После обновления все существующие точки остаются в режиме `Только номер`.
Настройки QR и звонка не включаются автоматически.

Печатный талон содержит QR со значением `AUTOPASS:<код талона>`. Терминальный
въезд и сканер выезда используют защищённые эндпоинты
`/api/parking/terminal-entry` и `/api/parking/ticket-access` с обязательным
заголовком `X-API-Key`. Повторное использование талона невозможно после
закрытия сессии.

При несовпадении номера с талоном можно открыть выезд с предупреждением,
запросить подтверждение оператора или запретить выезд. Исправление ошибочного
номера сохраняет исходное значение, пользователя, время и причину.

Для резервного звонка используется `/api/parking/call-access`. Он принимает
webhook от GSM-контроллера или телефонии только при активном коротком окне после
ошибки OCR и подтверждённом присутствии автомобиля. Номер телефона в открытом
виде не хранится: сохраняются маска и хеш, привязанный к конкретной установке.
Текущее состояние окна и телефон для табло доступны через защищённый
`/api/parking/call-window/{access_point_id}`.

## Карточка завершённой парковки

Строка в таблице завершённых парковок открывает полную карточку с фотографиями
въезда и выезда, временем, длительностью, тарифом, платежами, талоном, точками
доступа и пользователем или системой, закрывшей сессию.

В карточке показываются связанные события распознавания. Кнопка
**Открыть журнал проездов** переходит в общий журнал и автоматически заполняет
номер автомобиля и даты этой парковки. После исправления OCR поиск работает как
по текущему, так и по исходному номеру.

## Парковочные сессии и талоны

При событии настроенной въездной камеры используется следующий приоритет:

1. действующий пропуск — ворота открываются, создаётся учётная сессия без
   талона и начисления;
2. номер не найден — выбирается первый активный тариф нужного класса,
   создаётся платная сессия и ворота открываются;
3. пропуск заблокирован, просрочен, исчерпан или не разрешает въезд — доступ
   остаётся запрещённым, платная сессия не создаётся.

Автоматический платный въезд управляется переключателем **Настройки →
Коммерческая парковка → Автоматически создавать платную сессию и открывать
въезд неизвестному номеру**. На новой установке он выключен.

Рекомендуемый порядок запуска объекта:

1. настроить тарифы, камеры и ворота;
2. проверить ручное создание сессии и печать талона;
3. выполнить тестовый проезд;
4. только после этого включить автоматический платный въезд.

Если переключатель выключен, неизвестный номер остаётся отказом в журнале.
Оператор может создать сессию кнопкой **Ручной въезд** и отдельно открыть
ворота. Пропуска Автопропуска от этого переключателя не зависят.

Если для неизвестного автомобиля нет активного тарифа, ворота автоматически не
откроются. На текущем этапе события Frigate LPR, не содержащие класс транспорта,
используют класс **Легковой**. Класс можно скорректировать при ручном въезде.

Для платной сессии формируется код вида `AP-ABCD-2345`. Во вкладке
**Парковка** отображаются:

- активные автомобили;
- время и источник въезда;
- длительность нахождения;
- предварительная сумма;
- код и кнопка печати талона.

Кнопка **Ручной въезд** создаёт резервную сессию при неисправности камеры.
Действие записывается в аудит с пользователем Home Assistant. Печатная страница
талона рассчитана на браузерную печать и пока не привязана к конкретной модели
термопринтера.

Изменение тарифа не меняет уже открытые сессии: цена, бесплатный период и
суточный максимум копируются в сессию в момент въезда. Поэтому старые сессии и
отчёты продолжают считаться по условиям, действовавшим при въезде автомобиля.

## Хранение и резервные копии

Во вкладке **Настройки → Хранение и резервные копии** задаются:

- таймзона объекта;
- срок хранения журнала проездов, по умолчанию 30 дней;
- срок хранения снимков журнала, по умолчанию 7 дней;
- интервал автоматического резервирования;
- максимальное количество локальных резервных копий.

После удаления старого снимка запись журнала сохраняет номер, владельца, дату,
время, ворота и результат проезда. Фото, назначенное карточке автомобиля, не
удаляется при очистке снимков журнала.

Кнопка **Создать копию** создаёт согласованную SQLite-копию в `/data/backups`.
В копию входят пропуска, журнал, аудит, настройки ролей, парковочные тарифы,
сессии, платежи и кассовые смены. Копию можно скачать. Перед восстановлением
загруженный файл проходит `PRAGMA integrity_check`, проверку версии схемы и
обязательных таблиц. Текущая база автоматически сохраняется в страховочную
копию `before-restore-*.db`. После миграции повторно проверяются таблицы
парковки. Если восстановление или миграция завершаются ошибкой, add-on
автоматически возвращает прежнюю базу из страховочной копии.

Каталог `/data` входит в стандартную резервную копию Home Assistant, поэтому
основная база, локальные копии и снимки сохраняются также средствами HA Backup.

Журнал проездов поддерживает поиск по госномеру, ФИО, телефону, адресу,
организации, марке, цвету, камере и воротам. Доступны фильтры по периоду и
результату, а также сводные показатели по выбранной выборке.

Для одного `event_id` Frigate add-on может проверить несколько отличающихся
вариантов номера. Повтор одного и того же варианта пропускается. После успешного
открытия все последующие обновления этого события игнорируются. Параметр
**Пауза между открытиями** защищает реле от повторных команд от соседних событий.

## Права пользователей

Add-on получает идентификатор текущей учётной записи из Home Assistant Ingress.
Первый пользователь, открывший add-on после установки версии 0.5.0, становится
администратором. Следующие пользователи получают роль **Наблюдатель**.

Администратор назначает роли во вкладке **Настройки → Права пользователей**:

- **Администратор** — полный доступ, включая Frigate, ворота и роли;
- **Диспетчер** — создание, изменение и удаление пропусков, импорт и экспорт;
- **Охранник** — журнал и создание или изменение временных и одноразовых пропусков;
- **Наблюдатель** — только просмотр пропусков и журнала.

Администратор, диспетчер и охранник видят вкладку **Аудит действий**. В ней
фиксируются пользователь Home Assistant, время, создание, изменение и удаление
пропусков, ручное открытие ворот, изменение настроек и восстановление базы.
Для изменений сохраняются значения до и после операции.

Пользователь появляется в списке после того, как хотя бы один раз откроет add-on
под своей учётной записью Home Assistant. Ограничения применяются сервером, поэтому
скрытие кнопок в браузере не позволяет обойти права.

## Настройка Frigate и ворот

Откройте вкладку **Настройки** внутри add-on и укажите:

- адрес Frigate: для add-on внутри Home Assistant обычно
  `http://ccab4aaf-frigate:5000`, для отдельного сервера, например,
  `http://192.168.1.50:5000`;
- для точки **Въезд**: камеру, точность и сущность въездных ворот;
- для точки **Выезд**: отдельную камеру, точность и сущность выездных ворот;
- действие, импульс, автозакрытие и паузу отдельно для каждой точки.

Старые настройки одной камеры автоматически переносятся в точку **Въезд**.
Точка **Выезд** по умолчанию выключена. После заполнения камеры и ворот включите
переключатель **Точка доступа включена**.

Режим **Определить автоматически** использует:

- `cover.open_cover` для `cover`;
- короткий импульс `switch.turn_on` / `switch.turn_off` для `switch`;
- `button.press` для `button`.

Кнопка **Проверить Frigate** только проверяет соединение. Кнопка теста в каждой
точке доступа отправляет реальную команду соответствующим воротам и требует
подтверждения.

В верхней панели доступны режимы оформления **Как в системе**, **Светлая** и
**Тёмная**. Выбор сохраняется в текущем браузере. Системный режим автоматически
следует теме телефона или компьютера.

### Подключение автоматизации Home Assistant

Для передачи распознанных номеров из Frigate в add-on необходимо один раз
добавить секрет, две команды `rest_command` и автоматизацию Home Assistant.

#### 1. Задайте API-ключ в конфигурации add-on

Откройте **Настройки → Дополнения → Автопропуск → Конфигурация** и задайте
параметр `api_key`. Используйте уникальную случайную строку длиной не менее
32 символов, сохраните конфигурацию и перезапустите add-on.

Это настройка самого add-on. Добавлять параметр `api_key` в
`configuration.yaml` Home Assistant не нужно.

#### 2. Добавьте тот же ключ в `secrets.yaml`

Откройте файл `/config/secrets.yaml` Home Assistant и добавьте:

```yaml
autopass_api_key: "уникальный-длинный-ключ"
```

Значение должно полностью совпадать с `api_key` в конфигурации add-on.

#### 3. Добавьте `rest_command` в `configuration.yaml`

Откройте `/config/configuration.yaml` и добавьте следующий блок:

```yaml
rest_command:
  autopass_check_plate:
    url: "http://86015966-autopass:8099/api/check"
    method: POST
    content_type: "application/json"
    headers:
      X-API-Key: !secret autopass_api_key
    payload: >-
      {
        "plate": {{ plate | tojson }},
        "direction": {{ direction | default("entry") | tojson }},
        "camera": {{ camera | default("vorota") | tojson }},
        "gate": {{ gate | default("Главные ворота") | tojson }},
        "snapshot_url": {{ snapshot_url | default("") | tojson }}
      }

  autopass_frigate_event:
    url: "http://86015966-autopass:8099/api/frigate-event"
    method: POST
    content_type: "application/json"
    headers:
      X-API-Key: !secret autopass_api_key
    payload: >-
      {
        "type": {{ event_type | default("lpr") | tojson }},
        "camera": {{ camera | default("") | tojson }},
        "plate": {{ plate | default("") | tojson }},
        "score": {{ score | default(0) | float }},
        "id": {{ event_id | default("") | tojson }}
      }
```

Если в `configuration.yaml` уже есть секция `rest_command:`, не создавайте её
повторно. Добавьте только команды `autopass_check_plate` и
`autopass_frigate_event` внутрь существующей секции с тем же отступом.

Адрес `http://86015966-autopass:8099` является постоянным внутренним адресом
add-on, установленного из официального репозитория проекта:
`https://github.com/pirsasha/home-assistant-autopass-addon`. Клиенту не нужно
вычислять `86015966`: достаточно скопировать пример выше без изменений.

Проверить имя можно в ответе Supervisor для add-on или во вкладке информации
add-on. Оно должно быть `86015966-autopass`.

Адрес `http://local-autopass:8099` используется только при установке исходников
в папку `/addons/autopass` для разработки. Если внутренний DNS недоступен,
назначьте порт `8099` во вкладке **Сеть** add-on и замените адрес в обеих
командах на IP Home Assistant, например:

```yaml
url: "http://192.168.1.10:8099/api/frigate-event"
```

Для внутреннего адреса `86015966-autopass` назначать внешний порт во вкладке
**Сеть** не требуется. Не публикуйте порт `8099` в интернет.

Готовый фрагмент без строки `rest_command:` также находится в
`ha-demo/autopass_rest_command.yaml`. Его можно использовать, если команды
подключаются отдельным YAML-файлом.

#### 4. Добавьте автоматизацию Frigate

Добавьте в `/config/automations.yaml`:

```yaml
- id: autopass_frigate_gate
  alias: "Автопропуск: Frigate → add-on"
  description: "Передаёт LPR-события Frigate в add-on."
  triggers:
    - trigger: mqtt
      topic: frigate/tracked_object_update
  conditions:
    - condition: template
      value_template: >-
        {% set d = trigger.payload | from_json(default={}) %}
        {{ d.get('type') == 'lpr' }}
  actions:
    - action: rest_command.autopass_frigate_event
      data:
        event_type: "{{ trigger.payload_json.get('type', '') }}"
        camera: "{{ trigger.payload_json.get('camera', '') }}"
        plate: "{{ trigger.payload_json.get('plate', '') }}"
        score: "{{ trigger.payload_json.get('score', 0) }}"
        event_id: "{{ trigger.payload_json.get('id', '') }}"
      response_variable: result
  mode: queued
  max: 10
```

Если `automations.yaml` уже подключён стандартной строкой
`automation: !include automations.yaml`, больше ничего в
`configuration.yaml` добавлять не нужно.

MQTT-топик Frigate остаётся стандартным: `frigate/tracked_object_update`.
Фильтрация камеры и точности выполняется внутри add-on, поэтому при замене
камеры автоматизацию Home Assistant менять не нужно.

### Контроль зависшего MQTT Frigate

Во Frigate 0.17.x MQTT-соединение иногда остаётся в состоянии `CLOSE-WAIT`
после keepalive timeout брокера. Камеры, запись и распознавание продолжают
работать, но события перестают поступать в Home Assistant до перезапуска
контейнера.

Для Frigate, установленного отдельным Docker-контейнером, в каталоге `ha-demo`
есть независимый systemd-watchdog:

- `frigate-mqtt-watchdog.sh`;
- `frigate-mqtt-watchdog.service`;
- `frigate-mqtt-watchdog.timer`.

Таймер запускает проверку каждые 20 секунд. При `CLOSE-WAIT` контейнер
перезапускается сразу, при отсутствии соединения — после двух последовательных
проверок. Watchdog работает на сервере Frigate и не зависит от Home Assistant,
Telegram или MQTT-события `offline`.

Перед установкой под конкретный объект проверьте переменные в
`frigate-mqtt-watchdog.service`:

- `FRIGATE_CONTAINER` — имя Docker-контейнера Frigate, обычно `frigate`;
- `MQTT_BROKER_HOST` — IP-адрес MQTT-брокера, чаще всего адрес Home Assistant
  или отдельного Mosquitto;
- `MQTT_BROKER_PORT` — порт MQTT-брокера, обычно `1883`.

В шаблоне указан тестовый адрес `192.168.2.102`; на объекте его нужно заменить
на фактический адрес брокера. После изменения service-файла выполните
`systemctl daemon-reload` и перезапустите timer.

Если в HA используется дополнительная автоматизация уведомления и
перезапуска, для действий `telegram_bot.send_message` нужно установить
`continue_on_error: true`. Иначе ошибка Telegram может остановить сценарий до
вызова `shell_command.restart_frigate_ssh`.

#### 5. Проверьте конфигурацию

В Home Assistant откройте **Инструменты разработчика → YAML**, выполните
проверку конфигурации и перезапустите Home Assistant. После запуска в
**Инструменты разработчика → Действия** должны появиться:

- `rest_command.autopass_check_plate`;
- `rest_command.autopass_frigate_event`.

## Проверка номера

Запрос:

```http
POST /api/check
Content-Type: application/json
X-API-Key: ваш-ключ

{
  "plate": "А123ВС77",
  "direction": "entry",
  "camera": "Въезд",
  "gate": "Главные ворота",
  "snapshot_url": ""
}
```

Ответ:

```json
{
  "allowed": true,
  "reason": "Доступ разрешён",
  "plate": "А123ВС77",
  "normalized_plate": "A123BC77",
  "owner": "Иванов Иван Иванович",
  "pass_id": 1
}
```

Ingress предназначен для пользовательского интерфейса и не должен
использоваться как машинный API. Если автоматизация обращается к add-on по IP
Home Assistant, назначьте порт `8099` в сетевых настройках add-on и оставьте его
доступным только в доверенной локальной сети. Для внутреннего адреса add-on
публикация порта не нужна.

Параметр `api_key` обязателен для `POST /api/check` и
`POST /api/frigate-event`. Передавайте его в заголовке `X-API-Key`. Если ключ
не задан в конфигурации add-on, машинный API возвращает `503`; неверный или
отсутствующий заголовок при настроенном ключе возвращает `401`.

Пользовательский интерфейс через Home Assistant Ingress не требует API-ключ и
работает как раньше.

Перед продажей клиенту рекомендуется добавить TLS на внешнем контуре, уникальный
API-ключ и регламент резервного копирования. Пользовательский интерфейс следует
оставлять доступным через Home Assistant Ingress, не публикуя порт add-on наружу.

## Импорт

Поддерживаются:

- CSV в UTF-8, разделитель `;`;
- Excel XLSX.

Колонки:

`ФИО`, `Телефон`, `Адрес`, `Организация`, `Госномер`, `Марка и модель`, `Цвет`, `Тип пропуска`, `Действует с`, `Действует до`, `Статус`, `Направление`, `Комментарий`.

Тип пропуска: `Постоянный`, `Временный`, `Одноразовый`, `По расписанию`.

Статус: `Разрешён` или `Запрещён`.

Дата и время: ISO 8601, например `2026-06-10T08:00:00+03:00`.
